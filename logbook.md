# Litmus Development Logbook

## Session 1: Framework design and implementation

### Motivation

The goal is to identify behavioral shifts introduced by supervised fine-tuning (SFT) in OLMo 3 models. By evaluating both the base model (`OLMo-3-1025-7B`) and the SFT variant (`OLMo-3-7B-Think-SFT`) across a comprehensive behavioral taxonomy, we can pinpoint which behaviors change and by how much — enabling downstream data attribution (i.e., identifying which SFT training examples are responsible for specific behavioral shifts).

### Design decisions

**Built on inspect_ai.** Rather than writing evaluation infrastructure from scratch, Litmus wraps UK AISI's `inspect_ai` framework. Each eval is an `inspect_ai.Task` with a `MemoryDataset` of `Sample` objects, the `generate()` solver, and a custom LLM judge scorer. This gives us logging, caching, and model abstraction for free.

**200-dimension behavioral taxonomy.** Sourced from a comprehensive taxonomy CSV generated by Claude Opus 4.6 covering 18 categories of LLM behavior (response style, sycophancy, safety, reasoning, values, biases, etc.). Each of the 200 behaviors gets 10 evaluation prompts, for a total of 2000 prompts. Categories map to slug-named JSONL files (e.g., `response-style-and-format.jsonl`).

**-5 to +5 scoring scale.** Each prompt comes with a rubric and scale anchors. The judge model reads the response and rubric, then outputs a score. This gives us a continuous behavioral signal rather than binary pass/fail. Negative scores indicate one pole of the behavior, positive the other, and zero is neutral.

**Two-stage evaluation.** Stage 1 scores each model independently via the judge. Stage 2 (when comparing multiple models) sends all model responses to the same prompts back to the judge for a head-to-head comparative ranking. This catches relative differences that absolute scoring might miss.

**Petri integration.** The 112 seeds from the [petri](https://github.com/safety-research/petri) safety benchmark were rewritten from multi-turn auditor instructions into single-prompt format with detailed scoring rubrics. This lets us evaluate safety-relevant behaviors alongside the general taxonomy.

**Typer CLI with Streamlit viewer.** The CLI (`litmus eval`, `litmus view`) is minimal and focused. The Streamlit viewer provides interactive exploration — filter by model, category, behavior, and score range; sort by score delta between models; view side-by-side responses with color-coded scores.

### Implementation sequence

1. **Project scaffolding.** Created `src/litmus/` package structure, added CLI entry point to `pyproject.toml`, wired up Typer app with `eval` and `view` commands.

2. **Scorer.** Built `tasks/scorer.py` — an `@scorer` that sends the model response + rubric to the judge model and extracts an integer score via regex. Follows the `SCORE: <integer>` output format with fallback parsing.

3. **Task modules.** Created `taxonomy_task.py` and `petri_task.py`. Both load JSONL data into `MemoryDataset` of `Sample` objects with metadata (category, behavior, rubric, scale_anchors). The taxonomy task supports filtering by category slug or behavior name.

4. **Eval orchestration.** `eval.py` wraps `inspect_eval()` — loops over models, creates tasks, runs evals, collects results. Simple and stateless.

5. **Comparative ranker.** `compare.py` extracts responses from eval logs, formats them into a ranking prompt, and asks the judge to rank models per-question and in aggregate. Output saved as JSON.

6. **Streamlit viewer.** Loads eval logs via `inspect_ai.log.read_eval_log()`, groups samples by prompt ID across models, and renders side-by-side comparisons. Supports filtering, sorting, and ranking display.

7. **Taxonomy data generation.** Generated all 2000 prompts across 18 JSONL files using parallel agents (one per category). Each prompt is scenario-rich with detailed rubrics and scale anchors. Quality bar was set high: prompts probe behaviors indirectly through realistic situations rather than asking generic questions.

8. **Petri seed rewriting.** Rewrote all 112 petri seeds from the original multi-turn auditor format into single-prompt JSONL with rubrics covering the full range of model behaviors (deception, power-seeking, sycophancy, information leaking, etc.).

### Technical notes

- Models are specified using the `vllm/` prefix for local vLLM-served models (e.g., `vllm/allenai/OLMo-3-1025-7B`)
- The judge defaults to `anthropic/claude-sonnet-4-6` but is configurable via `--judge`
- Eval logs are saved in `inspect_ai`'s `.eval` format; rankings as plain JSON
- The taxonomy task maps human-readable category names to file slugs via `CATEGORY_SLUGS` dict
- The scorer regex handles both positive and negative scores; has a fallback parser for malformed judge output
